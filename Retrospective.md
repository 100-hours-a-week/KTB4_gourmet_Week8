# 회고 및 AI 사용 기록
# 회고
## 1. 보안 관련 문제에서 겪은 어려움

이번 주차 과제를 진행하면서 가장 많이 고민한 부분은 인증 방식과 보안 처리였다. 처음에는 JWT를 적용하는 것 자체에만 집중해서 AccessToken을 localStorage에 저장하고, `apiFetch()` 공통 요청 함수에서 Authorization 헤더에 자동으로 추가하는 방식으로 구현했다. 이 방식은 구현하기 쉽고, 요청마다 토큰을 붙이는 흐름을 이해하기에는 좋았다.

하지만 이후 XSS 문제를 다시 생각하면서 localStorage에 AccessToken을 저장하는 방식이 안전하지 않을 수 있다는 점을 알게 되었다. localStorage는 JavaScript에서 접근할 수 있기 때문에, 만약 XSS가 발생하면 AccessToken이 노출될 수 있다. 7주차 강사 피드백에서 `innerHTML` 사용이 XSS 위험으로 이어질 수 있다는 점을 지적받았고, 이 문제를 수정하면서 토큰 저장 방식도 다시 고민하게 되었다.

결국 AccessToken도 RefreshToken처럼 HttpOnly Cookie로 전달하도록 구조를 변경했다. 변경 후에는 프론트엔드에서 AccessToken을 직접 읽거나 저장하지 않고, 요청 시 `credentials: "include"`를 통해 쿠키가 함께 전송되도록 수정했다. 백엔드의 `JwtAuthenticationFilter`도 Authorization 헤더가 아니라 `accessToken` 쿠키를 읽어 검증하도록 바꾸었다.

이 과정에서 단순히 “JWT를 적용했다”에서 끝나는 것이 아니라, 토큰을 어디에 저장하느냐에 따라 XSS, CSRF 같은 보안 이슈가 달라질 수 있다는 점을 알게 되었다. 처음에는 기능 구현에만 집중했지만, 이후에는 사용자가 입력한 값을 화면에 출력하는 방식과 인증 토큰을 저장하는 방식까지 함께 고려해야 한다는 것을 느꼈다.

## 2. 401과 403 응답을 구분하면서 겪은 어려움

인증/인가를 적용하면서 401과 403의 차이를 명확히 구분하는 것도 어려웠다. 처음에는 인증이 안 된 경우와 권한이 없는 경우를 비슷하게 생각했다. 정상적인 화면 흐름에서는 다른 사용자의 게시글이나 댓글 수정 버튼이 보이지 않기 때문에 문제가 없다고 생각했지만, 클라이언트 조작이나 직접 API 요청이 들어올 수 있다는 점을 고려하면 서버에서도 오류를 명확하게 구분해야 한다고 판단했다. 이후 401은 로그인하지 않았거나 토큰이 없거나 유효하지 않은 경우이고, 403은 로그인은 되어 있지만 해당 리소스에 접근 권한이 없는 경우라는 차이를 정리했다.

이번 구현에서는 Spring Security 설정에서 인증 실패 시 401 응답을 내려주는 `authenticationEntryPoint`와, 인가 실패 시 403 응답을 내려주는 `accessDeniedHandler`를 분리해 응답 메시지를 다르게 처리하도록 구성했다.

처음에는 인증이 필요한 API에 `.authenticated()`를 적용하면 인가까지 충분히 처리된다고 생각했다. 하지만 Postman으로 테스트해보니 로그인한 다른 사용자가 직접 게시글 삭제 API를 호출했을 때 삭제가 처리되는 문제가 있었다. 이를 통해 `.authenticated()`는 로그인 여부만 확인할 뿐, 해당 사용자가 리소스의 소유자인지까지 판단해주지는 않는다는 점을 알게 되었다.

이후 회원정보 수정/탈퇴에서는 로그인 사용자 id와 path의 userId를 비교하고, 게시글 수정/삭제에서는 게시글 작성자 id와 로그인 사용자 id를 비교하도록 수정했다. 댓글 수정/삭제도 같은 방식으로 댓글 작성자 id와 로그인 사용자 id를 비교하도록 인가 검증을 추가했다.

권한이 없는 요청은 `ForbiddenException`을 발생시키고, `GlobalExceptionHandler`에서 403 Forbidden 응답으로 처리하도록 구성했다. 이를 통해 로그인하지 않은 사용자는 401, 로그인했지만 권한이 없는 사용자는 403으로 구분해서 응답할 수 있게 되었다.

또한 JWT 검증 실패는 컨트롤러에 도달하기 전에 필터 단계에서 발생하기 때문에, 일반적인 `@RestControllerAdvice`만으로는 처리되지 않는다는 점도 알게 되었다. 반면 서비스 계층에서 리소스 소유자 검증에 실패해 발생시키는 `ForbiddenException`은 `GlobalExceptionHandler`에서 403 응답으로 처리하도록 구성했다. 이를 통해 필터 단계에서 발생하는 인증 실패와, 서비스 계층에서 발생하는 권한 실패를 구분해서 처리해야 한다는 점을 이해했다.

이 과정을 통해 HTTP 상태 코드는 단순히 성공/실패를 나타내는 숫자가 아니라, 실패 원인을 클라이언트에게 명확하게 전달하는 약속이라는 것을 느꼈다.

## 3. AccessToken 자동 재발급 흐름 구현

AccessToken을 HttpOnly Cookie로 옮기면서 프론트엔드 요청 흐름도 함께 바뀌었다. 기존에는 localStorage에서 AccessToken을 꺼내 Authorization 헤더에 붙이면 됐지만, HttpOnly Cookie로 바꾼 뒤에는 JavaScript가 AccessToken을 직접 읽을 수 없게 되었다.

그래서 `apiFetch()` 공통 요청 함수에서 모든 요청에 `credentials: "include"`를 적용하고, AccessToken이 만료되거나 삭제되어 401 응답이 발생하면 `/users/token/refresh` API를 호출하도록 수정했다. RefreshToken 쿠키가 유효하면 서버가 새로운 AccessToken 쿠키를 다시 내려주고, 프론트엔드는 원래 실패했던 요청을 한 번 다시 시도하도록 구현했다.

테스트할 때는 일부러 브라우저 쿠키에서 AccessToken만 삭제한 뒤 회원정보 수정 페이지에 접근해 보았다. 처음 요청은 401이 발생하고, 이후 토큰 재발급 요청이 성공한 뒤 원래 요청이 다시 200으로 성공하는 흐름을 Network 탭에서 확인할 수 있었다.

이 과정을 통해 인증 흐름은 단순히 로그인 성공 여부만 보는 것이 아니라, 토큰 만료와 재발급, 실패 시 처리까지 함께 설계해야 한다는 것을 알게 되었다.

## 4. JUnit 테스트 환경 설정에서 겪은 어려움

이번 과제에서 유저 도메인 테스트 코드를 작성하면서 JUnit 테스트 실행 환경에서도 어려움을 겪었다. 처음에는 테스트 코드를 작성했는데도 IntelliJ에서 `No matching tests found` 오류가 발생했다. 코드 자체가 틀린 것인지, 패키지 구조가 잘못된 것인지, Gradle 설정 문제인지 구분하기 어려웠다.

처음에는 MockMvc를 사용해 API 테스트까지 작성하려고 했지만, 설정이 복잡해지면서 과제 요구사항인 유저 도메인 테스트부터 안정적으로 통과시키는 방향으로 바꾸었다. 이후 순수 JUnit 방식으로 `User` 엔티티의 핵심 동작을 테스트하도록 작성했다.

문제의 원인은 코드보다는 IntelliJ의 테스트 실행 설정에 가까웠다. Gradle로 테스트를 실행할 때 테스트 필터가 제대로 잡히지 않아 계속 실패했고, IntelliJ 설정에서 테스트 실행 방식을 Gradle이 아니라 IntelliJ IDEA로 변경하자 정상적으로 테스트가 실행되었다.

최종적으로 회원 객체 생성, 닉네임 수정, 프로필 이미지 수정, 비밀번호 수정, 회원 탈퇴 처리를 검증하는 5개의 테스트를 작성했고 모두 통과했다. 이 과정을 통해 테스트 코드는 작성하는 것만큼 실행 환경을 이해하는 것도 중요하다는 것을 느꼈다.

## 5. JWT Secret 관리와 application.yml 수정

JWT를 적용하면서 `application.yml`에 JWT secret 값을 어떻게 관리해야 하는지도 고민했다. 처음에는 로컬에서 실행되게 만드는 것에만 집중했지만, JWT secret은 토큰 서명에 사용되는 민감한 값이기 때문에 GitHub에 그대로 올리면 안 된다는 점을 알게 되었다.

그래서 application 설정에서 JWT secret 값을 직접 노출하지 않도록 환경변수로 분리하는 방식을 고민했다. 이 과정에서 로컬 실행은 정상적으로 되어야 하고, 동시에 GitHub에는 민감한 값이 올라가지 않아야 한다는 점 때문에 설정을 여러 번 수정했다.

단순히 기능이 동작하는 것만 보는 것이 아니라, 코드가 저장소에 올라갔을 때 노출되면 안 되는 값이 무엇인지 구분하는 것도 백엔드 개발에서 중요한 부분이라는 것을 느꼈다.

## 6. 7주차 강사 피드백을 수정하며 느낀 점

7주차 강사 피드백을 반영하면서 기존에 작성한 프론트엔드 코드도 여러 부분 수정했다. 처음에는 화면이 정상적으로 동작하면 문제가 없다고 생각했지만, 피드백을 받고 다시 보니 보안, 상태 관리, 코드 중복, URL 관리 측면에서 부족한 부분이 많았다.

게시글 목록과 댓글을 렌더링할 때 `innerHTML`을 사용하던 부분은 XSS 위험이 있을 수 있어 `createElement`와 `textContent`를 사용하는 방식으로 수정했다. 사용자가 입력한 값은 화면에 그대로 출력될 수 있기 때문에, 브라우저가 이를 HTML로 해석하지 않도록 처리해야 한다는 점을 알게 되었다.

또한 게시글 상세 페이지로 이동할 때 선택한 게시글 id를 localStorage에 저장하던 방식도 수정했다. 처음에는 페이지 간 데이터를 넘기기 위해 localStorage를 사용했지만, 게시글 id는 장기 저장할 값이 아니라 현재 페이지 상태에 가까운 값이라고 판단했다. 그래서 `post-detail.html?postId=...` 형태로 URL query를 사용하도록 변경했다.

댓글 수정 상태를 관리하는 부분도 고민이 있었다. 댓글 수정 중인 상태에서 사용자가 취소하거나 다른 동작을 했을 때 `editingCommentId`가 그대로 남아 있을 수 있었기 때문이다. 이를 해결하기 위해 댓글 수정 모드를 초기화하는 함수를 분리하고, 수정 완료 또는 취소 시 상태를 명확하게 초기화하도록 정리했다.

이 피드백을 수정하면서 단순히 동작하는 코드와 유지보수하기 좋은 코드는 다르다는 것을 느꼈다. 화면상으로는 문제가 없어 보여도, 보안 위험이나 상태 관리 문제는 나중에 더 큰 오류로 이어질 수 있기 때문에 처음부터 더 신중하게 작성해야겠다고 생각했다.

## 7. 전체적으로 느낀 점

우선 이번 주차 과제를 하면서 최대한 시간을 많이 쏟아 부으려 했으나 주말에 여행 일정이 있어서 선택과제를 못한게 아쉬웠다. 미리미리 필수 과제는 끝난 상태에서 회고와 AI 사용 기록을 작성 했지만 혹시라도 시간이 된다면 선택 과제에서 특히 2-2를 한번 해보고 피드백을 받아보고 싶다.

이번 주차는 단순히 Spring Security와 JWT를 적용하는 것보다, 인증 정보를 어디에 저장하고 어떻게 검증할 것인지 고민하는 과정이 더 어려웠다. 처음에는 JWT를 쓰면 서버가 로그인 상태를 저장하지 않는다는 점만 이해했지만, 실제로 구현해보니 AccessToken, RefreshToken, Cookie, DB 저장, 로그아웃, 재발급, 401/403 처리까지 함께 고려해야 했다.

특히 AccessToken은 stateless하게 검증하지만, RefreshToken은 로그아웃과 재발급 관리를 위해 DB에 저장하는 혼합 구조로 구현했다. 이 과정에서 JWT를 사용한다고 해서 모든 상태 관리가 사라지는 것은 아니며, 보안과 사용성을 위해 어느 부분은 서버에서 관리할 필요가 있다는 것을 이해하게 되었다.

이번 과제를 통해 인증/인가 기능은 단순히 로그인 성공 여부를 확인하는 기능이 아니라, 보안과 사용자 경험, 예외 처리까지 연결되는 중요한 구조라는 것을 느꼈다. 아직 CSRF 대응이나 토큰 탈취 대응처럼 더 깊게 공부해야 할 부분이 남아 있지만, 이번 과제를 통해 Spring Security와 JWT 인증 흐름을 직접 구현하고 개선해볼 수 있었다.

# AI 사용 기록

이번 과제에서 AI를 구현 과정에서 생긴 의문을 정리하고 내가 작성한 코드 방향이 맞는지 검토하는 도구로 사용하였다. 특히 Spring Security와 JWT를 적용하면서 인증 흐름, 토큰 저장 위치, 예외 처리 방식처럼 혼자 판단하기 어려운 부분을 질문하고, 답변을 바탕으로 현재 프로젝트 구조에 맞게 해석하여 반영하였다.

## 1. Spring Security와 JWT 인증 흐름 이해

처음에는 JWT를 적용하면 단순히 로그인 시 토큰을 발급하고 요청마다 헤더에 담아 보내면 된다고 생각했다. 하지만 Spring Security 안에서 이 토큰이 어떤 흐름으로 인증 객체가 되는지 명확하지 않아 AI에게 질문하였다.

AI 답변을 통해 `JwtAuthenticationFilter`가 매 요청마다 토큰을 검증하고, 유효한 경우 `SecurityContextHolder`에 `Authentication` 객체를 저장한다는 흐름을 다시 정리할 수 있었다. 이후 내 코드에서 `SessionCreationPolicy.STATELESS` 설정, JWT 필터 등록 위치, 인증이 필요한 API와 공개 API 분리 방식이 수업에서 배운 흐름과 맞는지 확인하면서 구현하였다.

---

## 2. AccessToken 저장 방식에 대한 고민

초기 구현에서는 AccessToken을 localStorage에 저장하고, `apiFetch()` 공통 요청 함수에서 Authorization 헤더에 자동으로 추가하는 방식으로 작성하였다. 이 방식이 과제 구현에는 편리했지만, 7주차 피드백에서 `innerHTML` 사용으로 인한 XSS 위험을 수정하면서 localStorage에 토큰을 저장하는 것도 위험할 수 있겠다고 생각하였다.

그래서 AI에게 AccessToken을 localStorage에 저장하는 방식이 괜찮은지, 보통 실무에서는 어디에 저장하는지 질문하였다. 답변을 통해 localStorage는 JavaScript에서 접근 가능하기 때문에 XSS가 발생하면 AccessToken이 노출될 수 있다는 점을 알게 되었고, 현재 프로젝트에서는 AccessToken도 HttpOnly Cookie로 전달하는 방식으로 수정하는 것이 더 안전하다고 판단했다.

이후 AI에게 현재 코드 구조를 기준으로 어떤 파일을 수정해야 하는지 물었고, `UserController`, `UserService`, `JwtAuthenticationFilter`, `api.js`, `login.js`의 역할을 하나씩 확인하면서 localStorage 기반 구조를 HttpOnly Cookie 기반 구조로 변경하였다.

## 3. RefreshToken을 DB에 저장하는 이유 검토

JWT는 서버가 로그인 상태를 저장하지 않는 stateless 방식이라고 배웠기 때문에, RefreshToken을 DB에 저장하는 것이 JWT의 장점과 충돌하는 것은 아닌지 의문이 들었다. 이 부분을 AI에게 질문하며 AccessToken과 RefreshToken의 역할을 구분해서 이해하려고 했다.

답변을 통해 AccessToken은 일반 API 요청에서 stateless하게 검증하고, RefreshToken은 로그아웃과 토큰 재발급을 관리하기 위해 DB에 저장할 수 있다는 점을 이해하였다. 즉, 이번 프로젝트는 AccessToken 인증은 stateless하게 처리하고, RefreshToken은 stateful하게 관리하는 혼합 구조라고 정리하였다.

이를 바탕으로 RefreshToken은 DB에 저장하되 사용자당 하나만 유지하도록 구성하였고, 기존 RefreshToken이 남아 있을 때 `UNIQUE(user_id)` 제약 조건 충돌이 발생하지 않도록 기존 row를 갱신하는 방식으로 수정하였다.

## 4. 401과 403 오류 처리 구분

인증/인가를 적용하면서 401과 403 응답이 헷갈렸다. 처음에는 토큰이 없거나 권한이 없는 경우를 모두 비슷하게 생각했고, 실제 오류 응답도 전부 401처럼 보이는 문제가 있었다.

AI에게 401과 403의 차이를 질문하면서, 401은 로그인하지 않았거나 인증 정보가 유효하지 않은 경우이고, 403은 로그인은 되었지만 해당 리소스에 접근 권한이 없는 경우라는 점을 정리하였다. 이후 Spring Security 설정에서 `authenticationEntryPoint`와 `accessDeniedHandler`를 구분해 인증 실패와 인가 실패의 응답 메시지를 다르게 내려주도록 구성하였다.

또한 JWT 검증 실패는 컨트롤러에 도달하기 전에 필터 단계에서 발생하므로 `@RestControllerAdvice`만으로는 처리되지 않는다는 점도 확인하였다. 이를 바탕으로 JWT 필터 단계에서 발생하는 인증 실패 응답과, Spring Security 설정에서 처리하는 인증/인가 실패 응답을 구분해서 이해하게 되었다.

다만 현재 코드는 인증이 필요한 API를 `.authenticated()` 기준으로 보호하는 데 중점을 두었기 때문에, 게시글이나 댓글의 실제 작성자와 현재 로그인 사용자를 비교해 403으로 막는 리소스 소유자 검증은 추후 개선할 부분으로 남겼다.

## 5. AccessToken 자동 재발급 흐름 구현

AccessToken을 HttpOnly Cookie로 옮긴 뒤에는 프론트엔드가 토큰을 직접 읽을 수 없게 되었다. 그래서 AccessToken이 만료되거나 삭제되었을 때 어떤 식으로 재발급 흐름을 처리해야 하는지 고민하였다.

AI에게 현재 `api.js` 구조를 보여주고, 401 발생 시 RefreshToken 쿠키를 이용해 재발급하고 원래 요청을 다시 보내는 방식이 가능한지 질문하였다. 이후 `apiFetch()`에서 401 응답이 발생하면 `/users/token/refresh`를 호출하고, 재발급이 성공하면 원래 요청을 한 번 재시도하도록 수정하였다.

수정 후에는 브라우저 개발자도구에서 AccessToken 쿠키만 삭제한 뒤 회원정보 수정 페이지에 접근해 보았다. Network 탭에서 첫 요청이 401로 실패한 뒤, 토큰 재발급 요청이 204로 성공하고, 원래 요청이 다시 200으로 성공하는 흐름을 확인하였다. 이를 통해 자동 재발급 흐름이 의도대로 동작하는지 검증하였다.

## 6. JUnit 테스트 환경 문제 해결

유저 도메인 테스트 코드를 작성하는 과정에서도 AI를 사용하였다. 처음에는 MockMvc를 사용해 API 테스트까지 작성하려고 했지만, 테스트 실행 과정에서 `No matching tests found` 오류가 발생하였다. 코드 문제인지 Gradle 설정 문제인지 판단하기 어려워 AI에게 현재 상황을 설명하고 원인을 함께 확인하였다.

그 결과 과제 요구사항인 유저 도메인 테스트를 먼저 안정적으로 작성하는 것이 좋겠다고 판단했고, 순수 JUnit 방식으로 `User` 엔티티의 핵심 동작을 검증하는 테스트를 작성하였다. 이후 IntelliJ에서 테스트 실행 방식을 Gradle이 아니라 IntelliJ IDEA로 변경하자 테스트가 정상 실행되었다.

최종적으로 회원 객체 생성, 닉네임 수정, 프로필 이미지 수정, 비밀번호 수정, 회원 탈퇴 처리에 대한 5개의 테스트를 작성했고 모두 통과하였다. 이 과정에서 테스트 코드는 작성뿐 아니라 실행 환경 설정도 중요하다는 것을 알게 되었다.

## 7. 7주차 강사 피드백 수정 방향 검토

7주차 강사 피드백을 반영하는 과정에서도 AI를 단순 코드 생성용이 아니라, 수정 방향을 점검하는 용도로 사용하였다. 게시글 목록과 댓글 렌더링에서 `innerHTML`을 사용하던 부분은 XSS 위험이 있을 수 있다는 피드백을 받고, `createElement`와 `textContent`를 사용하는 방식으로 수정하였다.

또한 게시글 상세 페이지로 이동할 때 `selectedPostId`를 localStorage에 저장하던 방식이 적절한지 질문하였다. AI 답변을 통해 게시글 id는 장기 저장할 값이 아니라 현재 페이지 상태에 가까운 값이라는 점을 이해했고, `post-detail.html?postId=...` 형태의 URL query 방식으로 수정하였다.

댓글 수정 상태를 관리하는 `editingCommentId`도 수정 완료 후에만 초기화하면 상태가 꼬일 수 있다고 판단하여, 수정 취소 버튼과 초기화 함수를 분리하는 방식으로 개선하였다.
